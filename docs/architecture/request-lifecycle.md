# 请求生命周期

客户端通过消费者密钥（`ska_`）调用 AI 网关的完整链路。

## 全链路概览

```
客户端
│  POST /api/gateway/ai/endpoint/v1/chat/completions
│  Authorization: Bearer ska_xxxxxxxx
│  Body: { model: "claude-opus-4-7", messages: [...], stream: true }
│
▼
┌──────────────────────────────────────────────────────┐
│  ① 请求 ID 中间件                                     │
│  读取 X-Request-ID 或生成 UUID                         │
│  注入 AsyncLocalStorage，所有日志自动携带                 │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ② 消费者密钥认证                                      │
│                                                      │
│  ska_ 密钥 → SHA-256 哈希 → 查询 relay_consumer_keys  │
│                                                      │
│  检查链:                                               │
│  ├── 密钥是否存在（不存在检查黑名单 → 403）               │
│  ├── 密钥状态 = active                                 │
│  ├── 关联用户未被禁用                                    │
│  ├── 未过期（expiresAt）                                │
│  ├── 关联 Pay Agent 存在且 active                      │
│  └── Pay Agent 余额 > 0（否则 402）                    │
│                                                      │
│  输出: ConsumerSession                                │
│  { consumerId, agentId, markupPercent,               │
│    allowedModels, perPayLimit, dailyLimit, ... }     │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ③ 请求体校验                                         │
│                                                      │
│  Zod 验证: model(必填), messages(≥1), stream,         │
│  temperature, max_tokens 等                           │
│  失败 → 400                                          │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ④ 模型 ACL 检查                                      │
│                                                      │
│  consumer.allowedModels 非空时，检查 model 是否匹配      │
│  支持尾部通配: "claude-*" 匹配 "claude-opus-4-7"        │
│  不匹配 → 403                                        │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑤ 输入护栏                                           │
│                                                      │
│  加载所有 enabled 的 ai_guardrail_configs              │
│  三种内置规则（无外部 API 调用）:                         │
│  ├── 关键词黑名单（正则匹配）                             │
│  ├── 单条消息长度限制                                    │
│  └── PII 检测（邮箱/电话/SSN/信用卡）                    │
│  命中 + action=block → 403                            │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑥ 模型路由解析                                       │
│                                                      │
│  查询 ai_model_routes (JOIN models + providers)       │
│  按 priority 分组，组内 weight 加权随机                  │
│  无路由 → 404                                        │
│                                                      │
│  输出: 有序候选列表                                     │
│  [{ route, provider, model }]                        │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑦ 语义缓存（仅非流式）                                 │
│                                                      │
│  SHA-256(model + messages) → LRU 内存 + Redis         │
│  命中 → 直接返回，不计费                                │
│  TTL: 5 分钟，容量: 10,000 条                          │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑧ 消费限额预检                                       │
│                                                      │
│  ├── 日限额: sumSpendingToday(agentId) → 超限 429     │
│  └── 月限额: sumSpendingThisMonth(agentId) → 超限 429 │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑨ 上游解析（对每条路由的每个上游）                       │
│                                                      │
│  对每条路由:                                           │
│  │                                                   │
│  ├─ providerModelId = route.providerModelId           │
│  │                    ?? model.modelId                │
│  ├─ adapter = getAdapter(provider.apiFormat)           │
│  │   (openai / anthropic / gemini / bedrock / azure)  │
│  ├─ transformRequest() → 转换为供应商原生格式             │
│  │                                                   │
│  └─ 对每个上游:                                        │
│     ├─ pickKey() → 加权轮询/随机选取 API 密钥            │
│     ├─ decrypt() → AES-256-GCM 解密密钥                │
│     ├─ resolveModelMapping() → 上游级模型名覆盖          │
│     ├─ adapter.buildUrl() → 构建上游 URL                │
│     └─ buildProviderAuth() → 构建认证头                 │
│        (Bearer / API-Key / SigV4 / Gemini query)     │
│                                                      │
│  输出: resolvedUpstreams[]                             │
│  无可用上游 → 403                                      │
└──────────────────────┬───────────────────────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑩ 上游请求 + 重试循环                                 │
│     最多 MAX_UPSTREAM_ATTEMPTS = 5 次                  │
│                                                      │
│  ┌─ 逐个尝试 resolvedUpstreams ──────────────────┐    │
│  │                                               │    │
│  │  fetch(finalUrl, { body, headers, timeout })  │    │
│  │                                               │    │
│  │  结果:                                        │    │
│  │  ├── 成功 → 进入响应处理                        │    │
│  │  ├── 可重试 (429/500/502/503/504)              │    │
│  │  │   → markKeyFailure() → 尝试下一个上游        │    │
│  │  └── 不可重试 (400/401/403)                    │    │
│  │      → 立即返回错误给客户端                      │    │
│  │                                               │    │
│  └───────────────────────────────────────────────┘    │
│                                                      │
│  全部失败 → 502 "All upstream candidates failed"      │
└──────────────────────┬───────────────────────────────┘
                       │
          ┌────────────┴────────────┐
          │                         │
    流式响应                    非流式响应
          │                         │
          ▼                         ▼
┌─────────────────────┐  ┌─────────────────────┐
│  ⑪a 流式处理         │  │  ⑪b 非流式处理       │
│                     │  │                     │
│  forwardStream()    │  │  解析 JSON 响应       │
│  SSE 逐帧转发        │  │  transformResponse() │
│                     │  │  extractUsage()      │
│  特性:              │  │                      │
│  ├ 空闲超时 5 分钟    │  │  计费:               │
│  ├ 硬超时 30 分钟    │  │  ├ 计算成本           │
│  ├ 心跳 15 秒/次    │  │  │  (tokens × price   │
│  └ 缓冲上限 1MB     │  │  │   × (1 + markup%)) │
│                     │  │  ├ 单次限额检查        │
│  流结束后异步计费:    │  │  ├ 日/月限额检查       │
│  billConsumer()     │  │  └ 扣款 + 记录交易     │
│  (响应已发送给客户端) │  │                      │
│                     │  │  写入语义缓存          │
│                     │  │  返回 c.json()        │
└─────────┬───────────┘  └──────────┬──────────┘
          │                         │
          └────────────┬────────────┘
                       │
┌──────────────────────▼───────────────────────────────┐
│  ⑫ 异步写入（不阻塞响应）                               │
│                                                      │
│  所有日志和计费通过 enqueueJob() 异步处理:               │
│                                                      │
│  ├── ai-usage-log    用量日志（批量 50条/1秒 刷盘）      │
│  ├── ai-request-log  请求体/响应体（按配置开关）          │
│  ├── agent-ai-txn    交易记录 + 密钥提供商分润           │
│  ├── ai-key-touch    更新密钥 last_used_at             │
│  └── consumer-key-touch  更新消费者密钥 last_used_at    │
│                                                      │
│  队列: BullMQ (Redis)，批量模式用内存缓冲               │
│  降级: Redis 不可用时静默丢弃 + 节流告警                  │
└──────────────────────────────────────────────────────┘
```

## 流式 vs 非流式差异

| 方面      | 流式                          | 非流式                   |
| --------- | ----------------------------- | ------------------------ |
| 响应方式  | SSE 逐帧转发                  | 一次性 JSON              |
| 计费时机  | 流结束后异步（响应已发送）    | 响应前同步（可拒绝）     |
| 单次限额  | 流后检查，超限挂起 Agent      | 响应前检查，超限返回 429 |
| 日/月限额 | 仅预检                        | 预检 + 响应前二次检查    |
| 语义缓存  | 不参与                        | 读 + 写                  |
| 超时      | 空闲 5min + 硬限 30min + 心跳 | 单一 30min               |

## 超时配置

| 参数                  | 默认值  | 说明                         |
| --------------------- | ------- | ---------------------------- |
| `upstreamFetchMs`     | 120 秒  | 等待上游首字节（含思考模型） |
| `streamIdleMs`        | 5 分钟  | 流式传输空闲断连             |
| `streamMaxDurationMs` | 30 分钟 | 流式传输硬性上限             |

可通过 Admin → Gateway Config 调整，存储于 `global_settings` 表。

## 可重试状态码

`429` `500` `502` `503` `504`

其他状态码（如 400/401/403）立即返回客户端，不重试。

## 密钥健康机制

- 成功: `markKeySuccess()` — 恢复权重
- 失败: `markKeyFailure()` — 指数惩罚（30 秒基准，2 分钟上限），降低有效权重
- 效果: 故障密钥自动降权，流量漂移到健康密钥
